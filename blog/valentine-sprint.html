February 14th, 2026. While most people were planning romantic dinners, we had our sights set on a different kind of love story ‚Äî the love for secure AI tooling. What started as a casual conversation about MCP security risks turned into the most intense 24-hour coding marathon of our careers.

The Spark

It all began during a late-night security discussion. We'd been watching AI agents proliferate across organizations, each one needing access to increasingly sensitive tools and data. The more we dug into how MCP (Model Context Protocol) servers were being deployed, the more concerned we became.

"We realized that giving AI agents direct access to our tools was like handing our car keys to a teenager and hoping for the best."

The problems were clear:

  • API keys scattered across config files, visible to AI agents
  • No audit trail of what AI agents were actually doing
  • Zero policy enforcement ‚Äî agents could access everything, always
  • Prompt injection attacks could bypass any application-level restrictions

By 2 AM, we had a whiteboard covered in diagrams. By 4 AM, we had a plan. By 6 AM, we had committed to building it in one day.

The Architecture

The core insight was simple: we needed a security proxy that could sit between AI agents and MCP servers, providing a policy layer without breaking the existing ecosystem. The agent thinks it's talking to a regular MCP server, but in reality, every request flows through our security engine.

We settled on a three-component architecture:

  • ToolGuard Proxy: The brain ‚Äî handles credentials, policies, logging, and anomaly detection
  • ToolGuard Shim: The adapter ‚Äî presents as a standard MCP server to the AI agent
  • Plugin System: Service-specific adapters for Google Workspace, Slack, GitHub, etc.

The Sprint

What followed was a blur of code, coffee, and constant iteration. We divided into three teams:

Team Proxy (8 AM - 2 PM)

Built the core proxy server with credential management and policy engine. The biggest challenge was designing a flexible policy language that could handle everything from rate limiting to content filtering without becoming too complex.

Team Shim (2 PM - 8 PM)

Created the MCP adapter that makes ToolGuard invisible to AI agents. This meant implementing the full MCP specification while transparently routing requests through our proxy. The devil was in the details ‚Äî handling streaming responses, error propagation, and maintaining perfect protocol compatibility.

Team Dashboard (8 PM - 12 AM)

Built a web interface for configuration and monitoring. Real-time audit logs, policy management, and anomaly detection dashboards. We wanted something that would give security teams immediate visibility into AI behavior.

The Breakthrough Moment

Around 10 PM, we had our first end-to-end test working. An AI agent successfully sent an email through Gmail, but instead of having direct access to the Gmail API, every request flowed through ToolGuard's policy engine. The agent never saw the OAuth tokens, the request was logged in full detail, and we could have blocked it based on recipient, content, or timing.

It was a simple test, but it proved the concept. We had built a security layer that was completely invisible to the AI agent but gave us total control over its actions.

Launch at Midnight

As February 14th turned into February 15th, we pushed the first version to GitHub. 24 hours from whiteboard to working code. The initial release included:

  • Core proxy and shim components
  • Google Workspace plugin (Gmail, Calendar, Drive)
  • Policy engine with rate limiting and access controls
  • Audit logging and real-time monitoring
  • Web dashboard for configuration

What We Learned

Building ToolGuard in one day taught us several important lessons:

1. Focus on the Core Problem

We could have spent weeks debating features, but the time constraint forced us to focus on solving the fundamental security issues. Everything else could wait.

2. Leverage Existing Standards

By building on MCP, we didn't have to create a new protocol or convince AI providers to integrate with us. We could provide security for the ecosystem that already exists.

3. Security Should Be Invisible

The best security tools are the ones that don't require developers to change their workflow. ToolGuard works because it's completely transparent to the AI agent.

What's Next

That Valentine's Day sprint was just the beginning. Since launch, we've added support for more services, improved the anomaly detection, and built enterprise features like SSO and advanced threat detection.

But the core principle remains the same: AI agents should be helpful, not dangerous. With ToolGuard, you can give your AI the tools it needs while maintaining complete control over what it can do.

"Sometimes the best way to build something is to just start building. ToolGuard exists because we stopped talking and started coding."

Looking back, February 14th, 2026 might not have been the most romantic Valentine's Day, but it was definitely one of the most productive. And in its own way, building secure AI tooling is a love letter to the future ‚Äî ensuring that as AI becomes more powerful, it remains trustworthy.
